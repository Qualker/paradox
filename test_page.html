Como funciona: Um POST com Content-Type: application/json obriga sempre a um preflight OPTIONS de CORS comum — isto acontece com ou sem rede local, é uma regra do browser há mais de uma década. O software local tem de responder com Access-Control-Allow-Origin (e afins) para o pedido real ser permitido. Adicionalmente, a partir do Chrome 142 existe também o Local Network Access (LNA): um popup de permissão pedido ao utilizador (não ao servidor) na primeira vez que um site acede à rede local — substituiu uma tentativa anterior ("Private Network Access") que exigia um header especial do servidor, entretanto abandonada pela Google. Isto significa que o servidor só precisa de responder bem a CORS comum; a permissão LNA é tratada inteiramente pelo browser e pelo utilizador.

📝 O que este PoC valida
✅ Sucesso significa: o preflight OPTIONS foi respondido com Access-Control-Allow-Origin (CORS comum) pelo software local, e o fetch de HTTPS (github.io) para HTTP local funcionou sem device-server no meio. Se o Chrome 142+ também mostrar um popup a pedir permissão de acesso à rede local (LNA) antes disso, tens de aceitar — mas essa permissão é pedida ao utilizador, não depende de nenhum header que o software local tenha de enviar.
⚠️ Limitações conhecidas: LNA é Chrome-only (Safari não suporta). O que mais frequentemente bloqueia o pedido é o software local não responder a CORS comum — se não devolver Access-Control-Allow-Origin (e afins) no preflight, o pedido falha mesmo que o software esteja a correr e a responder perfeitamente a clientes não-browser (curl, Postman). Usa local_server.py (mock, já implementa CORS correto) para confirmar primeiro que o teu fluxo funciona, depois aponta o campo "URL base" para o software real.
